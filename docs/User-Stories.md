# User Stories

This document defines the key user stories for the *Building AI with Simon* website. It describes how different users will interact with the site and what value they gain from it.

---

## 1. Primary User (Visitor / Recruiter / Developer)

### As a visitor, I want to understand the purpose of the website
- So that I can quickly grasp what “Building AI with Simon” is about

**Acceptance Criteria:**
- Clear hero section on the home page
- Vision & Mission is easily accessible from navigation
- Purpose is explained in under 10 seconds of reading

---

### As a visitor, I want to see the developer’s journey
- So that I can understand their learning progress and growth

**Acceptance Criteria:**
- Learning Journey page exists
- Entries are ordered chronologically
- Each entry shows what was learned and built

---

### As a visitor, I want to view completed and ongoing projects
- So that I can evaluate the developer’s skills and experience

**Acceptance Criteria:**
- Projects page lists all projects
- Each project includes description, tech stack, and GitHub link
- Project status is clearly indicated

---

### As a visitor, I want to understand the developer’s skills
- So that I can assess technical capability

**Acceptance Criteria:**
- Skills page is categorised (Frontend, Learning, Future)
- Skills are clearly readable and structured

---

### As a visitor, I want to contact the developer
- So that I can connect or collaborate

**Acceptance Criteria:**
- Contact page includes GitHub and social links
- Email or contact method is visible (if provided)
- No broken or missing links

---

## 2. Returning User (Follower of the Journey)

### As a returning user, I want to see updated progress
- So that I can track the developer’s improvement over time

**Acceptance Criteria:**
- Learning Journey is regularly updated
- New projects appear with timestamps or status changes

---

### As a returning user, I want consistent navigation
- So that I can easily move between pages

**Acceptance Criteria:**
- Navigation bar is consistent across all pages
- All links work correctly and are not broken

---

## 3. Developer (Site Owner - Simon)

### As the developer, I want to document my learning publicly
- So that I can build accountability and track progress

**Acceptance Criteria:**
- Learning Journey supports regular updates
- Content structure is easy to maintain manually

---

### As the developer, I want a simple static system
- So that I can focus on learning rather than backend complexity

**Acceptance Criteria:**
- No backend or database required
- Site runs fully on HTML and CSS (with minimal JS if needed)

---

### As the developer, I want to showcase my projects clearly
- So that my growth is visible to others

**Acceptance Criteria:**
- Projects page is easy to update
- GitHub links are properly connected
- Project structure remains consistent
